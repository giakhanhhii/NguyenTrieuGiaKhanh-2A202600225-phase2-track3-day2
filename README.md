# Lab #17: Multi-Memory Agent with LangGraph

This project implements a lightweight multi-memory agent aligned with the Lab #17 rubric:
- 4 memory types
- LangGraph state/router + prompt injection
- save/update logic with conflict handling
- benchmark on 10 multi-turn conversations
- privacy and limitations reflection

## Architecture
- `src/memory/short_term.py`: short-term conversation buffer
- `src/memory/profile.py`: long-term profile memory with Redis-first behavior and JSON fallback
- `src/memory/episodic.py`: episodic memory stored in JSON with lightweight retrieval
- `src/memory/semantic.py`: semantic memory with Chroma-first behavior and keyword fallback
- `src/router.py`: routes and aggregates relevant memory into state
- `src/prompting.py`: injects recent conversation, profile, episodic, and semantic sections into the prompt
- `src/context_window.py`: applies a simple prompt budget
- `src/agent.py`: LangGraph flow and local fallback runner
- `src/demo.py`: 3-session demo
- `src/benchmark.py`: benchmark runner

## Privacy and Limitations
### Privacy risk
Long-term profile memory is the most sensitive memory type because it can store persistent user facts such as name, allergies, and preferences across sessions.

### Risk controls
- Deletion: profile, episodic, and semantic memory are stored in separate backends/files, so each store can be deleted independently.
- TTL: Redis is the preferred backend for profile memory because TTL can be added later for time-bounded retention.
- Consent: persistent user memory should only be stored for information the user intentionally shares.
- Wrong retrieval risk: incorrect profile or episodic recall can lead to wrong advice, especially for preference or allergy facts.

### Current technical limitations
- The final response is rule-based instead of generated by a real LLM, so this lab focuses on memory architecture rather than model quality.
- Redis requires a running local server to be fully active; otherwise the project falls back to JSON persistence.
- Chroma is attempted first, but some local environments may block its runtime dependencies, so keyword fallback is kept as a safe backup.
